# Contributing

Workstream Continuity Design is an early research and design proposal. Contributions are most useful when they are bounded, evidence-backed, and tied to an implementation, study design, standards comparison, or concrete critique.

## Where to contribute

Use GitHub Discussions for open conceptual debate:

- category novelty and terminology;
- whether the nine-slot grammar is minimal, excessive, or missing a required slot;
- research lineage, adjacent standards, and crosswalks;
- reference implementation shape; and
- empirical validation methods.

Use GitHub Issues for concrete work:

- prototype tasks;
- study protocols;
- documentation corrections;
- accessibility or Markdown fixes;
- citation/source corrections; and
- narrowly scoped implementation experiments.

Use pull requests for specific repository changes. Keep PRs focused on one concern.

## Good contribution shape

A useful critique or proposal should state:

1. the claim, model, slot, pattern, or section being challenged;
2. the reason it may fail, over-specify, duplicate existing practice, or need support;
3. the evidence, implementation result, source, or test that motivates the concern; and
4. the smallest change or experiment that would resolve the question.

## Research status

This is a public, non-peer-reviewed research edition. The category terminology, continuity grammar, patterns, semantics policy, metrics, and maturity model are original synthesis and require empirical validation.

Please do not treat an accepted issue, discussion, or pull request as a standards decision. Durable changes should be reflected in the Markdown report, changelog, citation metadata when relevant, and future citable releases.
